# CLAUDE.md

Context for future Claude Code sessions working on this repo.

## What this is

Browser Pac-Man (vanilla JS + Canvas, no build step) with a persistent
SQLite leaderboard, served by a single Express app. Deploys as one Docker
container to Coolify.

## Stack

- Frontend: `public/index.html`, `public/game.js`, `public/style.css` --
  plain JS, no framework, no bundler. `game.js` is a single IIFE.
- Backend: `server.js` (Express) + `rateLimiter.js` (hand-rolled in-memory
  per-IP limiter, no dependency).
- DB: SQLite via `better-sqlite3`, one `scores` table.

## Run / build commands

```bash
npm install
npm start                 # http://localhost:3000

docker build -t pacman .
docker run -d -p 3000:3000 -v pacman_data:/data pacman
```

There is no test suite or build step -- `npm start` runs the server
directly, and the frontend is served as-is from `public/`.

## Env vars

- `PORT` -- default `3000`.
- `DB_PATH` -- default `./data/leaderboard.db` when unset (so local dev
  just works); the Dockerfile sets it to `/data/leaderboard.db` for
  container runs. The app creates the directory and table on first boot if
  they don't exist (`server.js`, `ensureDatabase()`).

## Deploy workflow (important -- keep this repo deploy-ready)

Coolify watches this GitHub repo and builds straight from the root
`Dockerfile` (build pack: Dockerfile). **Every push to `main` triggers an
automatic rebuild + redeploy** -- there's no separate CI gate, so `main`
should always be in a buildable, runnable state.

The leaderboard survives those redeploys because SQLite lives on a
persistent volume mounted at `/data` in Coolify (configured once in the
Coolify UI, not in this repo). The container itself is stateless; only
`/data` is durable. Coolify's health check hits `GET /healthz`.

Practical implications for changes here:
- Don't move the Dockerfile out of the repo root.
- Don't change the default `DB_PATH` container value away from
  `/data/leaderboard.db` without also updating the Coolify volume mount.
- Don't add a build step (bundler, TS compile, etc.) without updating the
  Dockerfile to run it -- Coolify only runs what the Dockerfile does.
- The SQLite file and `/data` must never be committed (see `.gitignore`).

## Game/maze notes

The maze in `public/game.js` (`MAZE` constant) was generated by a
recursive-backtracker + braid-pass algorithm (mirrored left/right for
symmetry) and flood-fill validated so every dot is reachable from Pac-Man's
start tile. If you resize or hand-edit the maze, re-validate connectivity
before shipping it -- a single dropped wall/dot can silently strand pellets
behind an unreachable wall, which only shows up as "the level never
completes."
